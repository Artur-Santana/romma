<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| email link → /auth/confirm | Token de convite (não confiável até validado pelo Supabase) cruza para o servidor |
| browser → /auth/reset-password | Nova senha digitada pelo usuário cruza para o Supabase Auth |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Elevation of Privilege | /auth/confirm (token replay) | mitigate | Supabase invalida o token_hash após o primeiro uso bem-sucedido; o handler não armazena nem reutiliza o token |
| T-07-02 | Spoofing | /auth/confirm (open redirect) | mitigate | Destino do redirect é hardcoded (`/portal/dashboard` no sucesso, `/login?error=invite_invalid` na falha) — não parametrizado por `?next=`/`?redirect=` |
| T-07-03 | Tampering | /auth/confirm (token expirado tratado como sucesso) | mitigate | Verificar `error` retornado por `verifyOtp`/`exchangeCodeForSession` ANTES de redirecionar para o portal; erro → /login?error=invite_invalid |
| T-07-04 | Information Disclosure | /auth/confirm (vazamento de motivo do erro) | accept | Erro genérico `invite_invalid` no querystring — não revela se o token expirou vs nunca existiu; risco baixo |
| T-07-05 | Tampering | /auth/reset-password (senha fraca aceita) | accept | Política de senha mínima é aplicada pelo Supabase Auth (server-side); validação client apenas confirma coincidência das senhas. Sem PII adicional exposta |
| T-07-SC | Tampering | npm/shadcn installs | mitigate | Nenhum pacote npm novo neste plano — `verifyOtp`/`updateUser` já disponíveis em @supabase/supabase-js 2.99.2 (VERIFIED em package.json) |
</threat_model>
