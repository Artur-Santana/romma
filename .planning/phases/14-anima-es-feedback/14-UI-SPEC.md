## Toast Contract

### Toaster Component Props (`src/app/layout.js`)

```jsx
<Toaster
  theme="dark"
  richColors
  position="bottom-right"
  mobileOffset={{ bottom: "80px" }}
  style={{
    "--normal-bg": "var(--surface)",
    "--normal-text": "var(--fg-1)",
    "--normal-border": "var(--border-2)",
    "--success-bg": "var(--surface)",
    "--success-text": "var(--fg-1)",
    "--success-border": "var(--success)",
    "--border-radius": "0px",
  }}
/>
```

**Design decisions:**

| Decision | Value | Rationale |
|----------|-------|-----------|
| `richColors` | `true` | Sonner only reads `--success-bg/-text/-border` when `richColors` is active (`[data-rich-colors="true"]`). Setting `richColors={false}` would make those three overrides dead code and yield only the default green checkmark icon with no border contrast. `richColors={true}` + explicit `--success-*` overrides applies the tamed muted-dark appearance instead of sonner's default saturated green. |
| `theme` | `"dark"` | Prevents sonner's light-mode defaults from flashing before CSS var overrides apply |
| `--border-radius` | `"0px"` | Design system uses `--radius: 0` (sharp corners). Sonner default is `8px` — must override explicitly |
| `--normal-bg` | `var(--surface)` | `oklch(0.182 0 0)` — matches dashboard card/surface tone |
| `--normal-text` | `var(--fg-1)` | `oklch(1 0 0)` — full white, matches primary text |
| `--normal-border` | `var(--border-2)` | `oklch(1 0 0 / 0.10)` — subtle border consistent with existing dashboard borders |
| `--success-bg` | `var(--surface)` | Same dark surface, not saturated green — consistent with Obsidian muted theme |
| `--success-text` | `var(--fg-1)` | Same white text — success is signaled by the border accent |
| `--success-border` | `var(--success)` | `oklch(0.696 0.17 162.5)` — the existing project success green token, tames sonner's default |

### Toast Messages (locked from D-08)

| Trigger | Call | Message |
|---------|------|---------|
| Criar contrato (success) | `toast.success("Contrato criado")` | "Contrato criado" |
| Encerrar contrato (success) | `toast.success("Contrato encerrado")` | "Contrato encerrado" |
| Cancelar contrato (success) | `toast.success("Contrato cancelado")` | "Contrato cancelado" |
| Deletar unidade (success) | `toast.success("Unidade removida")` | "Unidade removida" |
| Revogar acesso (success) | `toast.success("Acesso revogado")` | "Acesso revogado" |
| Marcar parcela como paga (success) | `toast.success("Parcela marcada como paga")` | "Parcela marcada como paga" |

Note: RESEARCH Open Question Q1 is resolved. "Unidade removida" is the message for `deletarUnidade`, consistent with D-05 (Toast ✅) and D-08 (full message list).

**Scope:** `toast.success()` only. No `toast.error()` in this phase — errors stay on inline `erro` state. Toast fires only after `result.status === 200`.

### Toast Duration

Default sonner duration (4000ms). No override — no `duration` prop on `<Toaster>` and no per-call `{ duration }` option. The default suits all 6 messages.

### Mobile Offset

The MobileBottomNav occupies approximately 62px at the bottom of the viewport on screens <640px (`pt-[14px]` + `pb-4` = 30px vertical padding + ~26px for label + code text rows = ~62px rendered height).

**Resolved (not deferred):** `mobileOffset={{ bottom: "80px" }}` on `<Toaster>`. This provides the 62px nav height + 18px clearance gap. Toast position remains `bottom-right` on all viewports; the offset clears MobileBottomNav on mobile without requiring a separate position prop.

## Exit-Animation Contract

### CSS Inline Style Pattern

Applied to the wrapper element of each list item during removal:

```js
style={{
  opacity: isRemoving ? 0 : 1,
  transform: isRemoving ? "scale(0.97)" : "scale(1)",
  transition: "opacity 200ms ease, transform 200ms ease",
}}
```

| Property | Resting value | Removing value | Notes |
|----------|--------------|----------------|-------|
| `opacity` | `1` | `0` | Primary exit signal |
| `transform` | `scale(1)` | `scale(0.97)` | Slight shrink — reinforces removal without dramatic layout shift |
| `transition` | — | `opacity 200ms ease, transform 200ms ease` | Always present so reverse (rollback) is also animated |
| `duration` | — | 200ms | Matches ANIM-01 spec (`~200ms`) |
| `easing` | — | `ease` (cubic-bezier(0.25, 0.1, 0.25, 1.0)) | Standard ease — sufficient at 200ms |

**Note on existing `.romma-unit-out` keyframe:** `globals.css` already declares `.romma-unit-out` (600ms, `translateX(40px)`, `blur(4px)`). This is a **different, pre-existing pattern** for a distinct use case. This phase intentionally uses lighter inline CSS per D-02 — no class assignment. The checker/auditor should not flag this as inconsistency.

### State Pattern

```js
const [removingIds, setRemovingIds] = useState(new Set())
```

One `removingIds` Set per component. Controls the `isRemoving` boolean in each list item's style.

### Timing Sequence

1. User confirms action → `setRemovingIds(prev => new Set([...prev, id]))` (animation starts)
2. `await serverAction(id)` → if error, rollback: `setRemovingIds(prev => { const n = new Set(prev); n.delete(id); return n })`
3. `result.status === 200` → `toast.success("message")` fires immediately (does not wait for animation)
4. `setTimeout(() => { removeFromArray(); clearRemovingId() }, 200)` → item removed from DOM after animation completes

**Exception for Contratos (cancelar/encerrar):** Server Action is UPDATE (row persists in DB). `getContratos()` returns all statuses — re-fetch would bring the item back. Use optimistic filter: `setContratos(prev => prev.filter(c => c.id !== id))` inside the `setTimeout`. Do not re-fetch contratos. Do re-fetch unidades (status changed).

### Components Receiving Exit Animation

| Component | Action | Animation type |
|-----------|--------|----------------|
| `Contratos.js` | `confirmarCancelamento` | Optimistic filter (no re-fetch) |
| `Contratos.js` | `confirmarEncerramento` | Optimistic filter (no re-fetch) |
| `Unidades.js` | `handleDeletarUnidade` | Re-fetch after setTimeout (DELETE) |
| `LocatariosDesktop.js` | `handleRevogar` | Re-fetch after setTimeout (DELETE) |
| `Locatarios.js` | `handleDeletarLocatario` | Re-fetch after setTimeout (DELETE) |
| `Parcelas.js` | `marcarComoPaga` | **No animation** — item stays in list with new status |
