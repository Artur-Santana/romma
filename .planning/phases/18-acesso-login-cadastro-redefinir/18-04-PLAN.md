<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Supabase Auth (resetPasswordForEmail / updateUser) | Untrusted email + recovery-session password change |
| Email recovery link → /auth/confirm → recovery session | One-time token grants temporary password-change capability |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-11 | Spoofing | recovery token replay | mitigate | verifyOtp (in /auth/confirm) invalidates the one-time recovery token; updateUser requires the live recovery session cookie. No change weakens this. |
| T-18-12 | Elevation of Privilege | wrong-role redirect after reset | mitigate | Replace unconditional /portal/dashboard with `rpc("is_proprietario")` branch (RESEARCH Pitfall 2) — proprietário lands on /dashboard, locatário on /portal/dashboard; redirect is post-auth and does not grant access by itself (RLS still enforces). |
| T-18-13 | Information Disclosure | reset request reveals account existence | accept | resetPasswordForEmail returns success regardless of whether the email exists (Supabase default) — the success banner is shown uniformly, no enumeration signal. |
| T-18-14 | Tampering | weak new password via define-flow | mitigate | validarSenha (tested, Plan 02) blocks <6/no-uppercase/no-number before updateUser; Supabase enforces ≥6 server-side as backstop. |
| T-18-SC | Tampering | npm installs | mitigate | No package installs this phase (RESEARCH audit: none). |
</threat_model>
