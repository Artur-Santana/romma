<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Server Action (`cadastrarProprietario`) | Untrusted signup fields cross into server-side signUp |
| Email link → Route Handler (`/auth/confirm`) | Untrusted token_hash/code from email crosses into verifyOtp + admin INSERT |
| Route Handler → Postgres (supabaseAdmin) | service-role write into `proprietarios` bypasses RLS |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-01 | Tampering | signUp options.data metadata | accept | Stored as JSONB in auth.users.raw_user_meta_data; no SQL surface; Supabase sanitizes. Values only land in proprietarios via parameterized supabase-js insert. |
| T-18-02 | Elevation of Privilege | supabaseAdmin in confirm/route.js | mitigate | SUPABASE_ROLE_KEY stays server-only (route handler + `server-only` guard on supabaseAdmin import); never imported by components in this plan. |
| T-18-03 | Spoofing | recovery/signup OTP token replay | mitigate | verifyOtp invalidates the one-time token (Supabase default); confirm route does not re-accept used tokens. No change to that behavior. |
| T-18-04 | Information Disclosure | empty metadata → null proprietarios rows | mitigate | Read user_metadata only from verifyOtp response (Pitfall 1); columns are nullable so a re-confirm with missing meta does not crash, but normal flow populates all three. |
| T-18-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this phase (RESEARCH Package Legitimacy Audit: none). No checkpoint required. |
</threat_model>
