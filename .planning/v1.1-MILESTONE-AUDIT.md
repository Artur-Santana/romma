# v1.1 Polish & Completeness — Milestone Audit

**Status:** gaps_found (2 gaps parciais, ambos documentáveis/aceitáveis p/ banca single-instance)
**Audited:** 2026-06-12 · banca 18/06 (6 dias)

## Resumo

8/8 fases verificadas. Integração cross-phase **sólida**: 28 conexões wired, 0 fluxos quebrados, 5 jornadas E2E completas (visitante→/unidades, signup→dashboard, CRUD→contrato→parcelas, convite locatário→portal, dashboard mobile). 27 unit + 73 E2E passando em CI (PR #34 verde).

## Gaps (2 parciais)

### AUTH-02 — guard de instância única só DB-side
2ª tentativa de signup no formulário não é bloqueada com mensagem; mostra `email_sent`. Se confirmar email, cria 2º proprietário. **Aceitável** sob premissa single-instance do TCC, mas a rota `/signup` é alcançável por qualquer um que saiba a URL. **Risco de banca: ALTO** — apresentar /signup como "setup inicial", garantir Supabase limpo antes da demo.

### MT-02 — 4 write Actions sem scoping de proprietario_id
`criarUnidade`, `criarContrato`, `editarContrato`, `marcarParcelaComoPaga` usam `supabaseAdmin` (bypassa RLS) sem ownership pre-check. Protegidas por `authGuard` (auth+is_proprietario) mas não por tenant. Não-exploráveis em demo single-instance; IDOR real em multi-tenant. MT-02 literal ("todas") não 100% cumprido.

## Tech debt (não-bloqueador)
- Locatarios.js orphaned (sem impacto — LocatariosDesktop é a rota real)
- THEME-02 paletas sem UI toggle
- Header nav placeholders href='#' (risco A1 aceito)

## Veredito
Nenhum gap é bloqueador funcional para demo single-instance. Ambos os gaps parciais são de **segurança multi-tenant** — relevantes se a banca sondar isolamento, irrelevantes para o fluxo demonstrado. Decisão do dono: fechar IDOR restante (pequeno, espelha 15-02) ou aceitar como deferido documentado.
